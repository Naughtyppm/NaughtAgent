# Iterative-Probe 升级计划：自主开发能力

## 目标

将 iterative-probe 从"执行者"升级为"开发者"，具备 AutoGPT 式的自主开发循环能力。

## 核心差距（精简版）

| 能力 | 当前 | 目标 |
|------|------|------|
| 输入 | "验证 DT→DA 链路" | "实现 XX 功能" |
| 规划 | 用户定义检查点 | 自动分解任务 |
| 决策 | 遇到分支问用户 | 自主选择最优方案 |
| 闭环 | 探测→修复→验证 | 规划→开发→测试→修复→完成 |

---

## 升级方案

### 1. 自主规划引擎

**触发**：用户给出模糊目标时自动启动

```
用户: "实现用户登录功能"
      │
      ▼
┌─────────────────────────────────────┐
│           目标分析                   │
│  - 这是什么类型的任务？              │
│  - 需要哪些组件？                    │
│  - 有哪些依赖？                      │
└─────────────────────────────────────┘
      │
      ▼
┌─────────────────────────────────────┐
│           任务分解                   │
│  1. [分析] 理解现有代码结构          │
│  2. [设计] 确定实现方案              │
│  3. [开发] 编写代码                  │
│  4. [测试] 验证功能                  │
│  5. [修复] 处理问题                  │
└─────────────────────────────────────┘
      │
      ▼
┌─────────────────────────────────────┐
│           执行计划                   │
│  按依赖顺序执行，遇到问题自主解决    │
└─────────────────────────────────────┘
```

**实现**：在 skill 中添加"规划阶段"模板

```markdown
## Phase 0: 自主规划（新增）

收到开发任务后，先执行规划：

### 步骤 1：任务分类
判断任务类型：
- 新功能开发
- Bug 修复
- 重构优化
- 系统集成

### 步骤 2：自动分解
根据任务类型生成子任务：

**新功能开发**：
1. 分析需求 → 理解要做什么
2. 探索代码 → 找到插入点
3. 设计方案 → 确定怎么做
4. 实现代码 → 写代码
5. 编写测试 → 验证正确性
6. 集成验证 → 确保不破坏现有功能

**Bug 修复**：
1. 复现问题 → 确认症状
2. 定位根因 → 找到问题代码
3. 设计修复 → 确定方案
4. 实现修复 → 改代码
5. 验证修复 → 确认问题解决
6. 回归测试 → 确保没引入新问题

### 步骤 3：输出计划
```markdown
## 开发计划

### 目标
{用户目标}

### 任务分解
| # | 任务 | 类型 | 依赖 |
|---|------|------|------|
| 1 | {任务1} | 分析 | - |
| 2 | {任务2} | 开发 | 1 |
| ... | ... | ... | ... |

### 执行策略
- 串行/并行说明
- 风险点预判
```

**不等用户确认，直接开始执行！**
```

### 2. 自主决策机制

**原则**：能自己决定的不问用户

```markdown
## 决策矩阵

### 自主决策（不问用户）

| 场景 | 决策方式 |
|------|----------|
| 多种实现方案 | 选择最简单/最常见的 |
| 命名选择 | 遵循项目现有风格 |
| 文件位置 | 遵循项目现有结构 |
| 技术选型 | 优先使用项目已有依赖 |
| 错误处理 | 先尝试修复，失败再换方案 |
| 测试范围 | 覆盖主要路径 + 边界情况 |

### 必须问用户

| 场景 | 原因 |
|------|------|
| 需求不明确 | 无法判断用户真正想要什么 |
| 破坏性变更 | 删除代码、改变接口 |
| 外部依赖 | 需要安装新包、访问外部服务 |
| 超出范围 | 发现任务比预期大很多 |

### 决策记录

每次自主决策后记录：
```markdown
### 决策 #{N}
- **问题**：{需要决策的问题}
- **选项**：A / B / C
- **选择**：A
- **理由**：{为什么选 A}
```

最终报告中汇总所有决策，让用户知道做了什么选择。
```

### 4. 更新后的 Skill 结构

```markdown
# 自主开发模式（升级版 iterative-probe）

## 会话标识

触发后，每次回复开头使用：
```
🎯 【自主开发】
```

## 执行流程

### Phase 0: 规划（新增）
1. 分析用户目标
2. 自动分解任务
3. 确定执行顺序
4. **直接开始执行，不等确认**

### Phase 1: 开发
1. 按计划执行每个任务
2. 遇到决策点自主选择
3. 记录所有决策

### Phase 2: 测试
1. 运行相关测试
2. 手动验证关键路径
3. 检查是否引入问题

### Phase 3: 修复
1. 分析测试失败原因
2. 修复问题
3. 重新测试
4. 循环直到通过

### Phase 4: 完成
1. 汇总所有变更
2. 列出所有决策
3. 说明验证结果

## 自驱规则（继承）

- 禁止单轮停止
- 禁止浅尝辄止
- 禁止被动等待
- 必须闭环验证

## 自主决策规则（新增）

- 多方案时选最简单的
- 遵循项目现有风格
- 先尝试再问用户
- 记录所有决策

## 终止条件

1. ✅ 功能完成 + 测试通过
2. 🚫 遇到必须问用户的情况
3. ⚠️ 5 轮修复仍未通过
```

## 实施步骤

### Step 1: 修改 skill.md

在现有 iterative-probe 基础上添加：

1. **Phase 0: 规划** - 任务分解逻辑
2. **决策矩阵** - 什么自己决定，什么问用户
3. **开发循环** - 替换探测循环

### Step 2: 添加任务模板

```markdown
## 任务类型模板

### 新功能开发
1. 探索 → 找到相关代码
2. 设计 → 确定方案
3. 实现 → 写代码
4. 测试 → 验证
5. 完成 → 汇报

### Bug 修复
1. 复现 → 确认问题
2. 定位 → 找根因
3. 修复 → 改代码
4. 验证 → 确认修复
5. 回归 → 检查副作用

### 重构
1. 分析 → 理解现状
2. 设计 → 确定目标
3. 重构 → 改代码
4. 测试 → 确保不破坏
5. 完成 → 汇报
```

### Step 3: 测试验证

用以下场景测试：

1. "给这个项目添加一个配置文件热重载功能"
2. "修复 MCP 连接超时的问题"
3. "重构 tool registry，支持动态注册"
