### 3.4 四种执行模式详解

| 模式 | 用途 | 工具调用 | 上下文 | 复杂度 |
|------|------|----------|--------|--------|
| `ask_llm` | 简单问答 | ❌ | 无状态 | 最低 |
| `run_agent` | 独立任务 | ✅ | 全新 | 中 |
| `fork_agent` | 子任务 | ✅ | 继承父 | 中高 |
| `run_workflow` | 复杂流程 | ✅ | 编排 | 高 |

**执行模式选择流程**：
```
需要执行任务
    │
    ├─ 不需要工具 ──────► ask_llm (简单问答)
    │
    └─ 需要工具
           │
           ├─ 不需要父上下文
           │      │
           │      ├─ 简单流程 ──► run_agent (独立任务)
           │      │
           │      └─ 复杂流程 ──► run_workflow (工作流)
           │
           └─ 需要父上下文 ──────► fork_agent (子任务)
```

### 3.5 MCP 协议集成

**MCP (Model Context Protocol)** 是 Anthropic 定义的工具协议标准：

```typescript
// MCP 工具定义
interface MCPTool {
  name: string;
  description: string;
  inputSchema: JSONSchema;    // 输入参数 schema
  outputSchema?: JSONSchema;  // 输出结果 schema (可选)
}

// MCP 客户端流程
1. connect()        → 连接 MCP 服务器
2. tools/list       → 获取可用工具列表
3. tools/call       → 调用工具
4. tools/list_changed → 监听工具变化
```

**传输方式**：
- **stdio**：通过标准输入输出通信（本地进程）
- **SSE**：Server-Sent Events（远程服务）

### 3.7 核心设计理念

**1. 单线程主循环（Single-Threaded Master Loop）**
```typescript
while (true) {
  const response = await claude.chat(messages);
  if (response.hasToolCalls()) {
    const results = await executeTools(response.toolCalls);
    messages.push(results);
    continue;
  } else {
    return response.text;  // 循环结束
  }
}
```

**2. 实时引导队列（H2A Queue）**
- 支持暂停/恢复执行
- 支持中途注入新指令
- 双缓冲异步消息队列

**3. 多层安全防护（Swiss Cheese Defense）**
- 模型对齐层：Claude 训练时的安全意识
- 权限回调层：canUseTool 回调
- 命令解析层：Bash 注入检测
- 沙箱隔离层：容器级别隔离

### 3.8 OpenCode 与 Claude Agent SDK 对比

| 维度 | OpenCode | Claude Agent SDK |
|------|----------|------------------|
| 语言 | Go | TypeScript |
| 执行模式 | 单一循环 | 4 种模式 |
| 工具协议 | 自定义 | MCP 标准 |
| 流式响应 | 基础支持 | 完整规范 |
| 子代理 | 无 | fork_agent |
| 工作流 | 无 | run_workflow |

### 3.9 优势与局限

| 优势 | 局限 |
|------|------|
| 架构简洁，易于理解 | 单线程，无法并行执行 |
| 实时可控，支持中途调整 | 依赖 Claude 模型 |
| MCP 生态，可扩展性强 | 上下文窗口限制 |
| 多层安全，防护完善 | 学习曲线（4种模式） |
| 官方规范，持续演进 | SDK 仍在发展中 |

### 4.3 核心特性

**1. 三种交互模式**

| 模式 | 功能 | 自主程度 |
|------|------|----------|
| Chat | 问答、解释代码 | 低 |
| Composer | 多文件编辑 | 中 |
| Agent | 自主完成任务 | 高 |

**2. 上下文引擎**
- **代码库索引**：全量向量化，支持语义搜索
- **符号解析**：跨文件跳转、引用分析
- **依赖图**：理解模块间关系
- **Git 历史**：理解代码演进

**3. Diff-based 编辑**
- 所有修改以 Diff 形式展示
- 用户可逐行接受/拒绝
- 支持部分应用

### 4.4 优势与局限

| 优势 | 局限 |
|------|------|
| IDE 深度集成，体验流畅 | 闭源，无法自定义 |
| 上下文理解强（向量索引） | 订阅制，成本较高 |
| 多模式适应不同场景 | 依赖云端服务 |
| Diff 预览，修改可控 | 大型项目索引慢 |

## 11. Agent-Wechat (Swarm-IDE) 架构分析

> 调研时间：2025-02
> 项目地址：https://github.com/chmod777john/agent-wechat

### 11.1 基本信息

| 属性 | 值 |
|------|-----|
| 项目名 | Swarm-IDE: 自组织的 Agent 蜂群 |
| 开发者 | chmod777john |
| 语言 | TypeScript |
| 开源 | ✅ |
| 定位 | 多 Agent 协作系统（微信式交互） |

### 11.2 核心定位

一个基于「微信式聊天界面」的多 Agent 协作系统，支持：
- 动态创建子 Agent（蜂群自组织）
- Agent 间任意通信
- 人类可随时介入任意层级的 Agent 对话
- 实时可视化 Agent 协作拓扑

### 11.3 设计哲学

| 原则 | 说明 |
|------|------|
| **极简原语** | 系统只依赖 `create` + `send` 两个核心原语，复杂协作由此组合 |
| **液态拓扑** | 拓扑不预设，运行时自演化；Agent 主动"雇佣"下属 |
| **扁平协作** | 人类可像聊天一样介入任意层级，使复杂拓扑可观察、可调试、可介入 |
| **去中心化** | 用户和 Agent 完全等价，用户只是一种特殊的 Agent |

**核心概念**：
> 没有 nodes 和 edges 的复杂抽象，只需把系统理解为"很多个人"：
> **每个人都能生孩子、也能和任意一个人说话。**
> 只要有这两种能力，就能实现任意结构。

### 11.4 技术栈

| 层级 | 技术选型 |
|------|----------|
| **Runtime** | Bun |
| **Backend** | Next.js 16 (Route Handlers) |
| **ORM** | Drizzle ORM (PostgreSQL) |
| **Streaming** | Redis Streams + SSE |
| **Frontend** | Next.js + React 19 + Tailwind 4 + Framer Motion |
| **LLM** | 智谱 GLM-4.7 / OpenRouter (Kimi 2.5) |
| **扩展** | MCP 协议支持 |

### 11.5 核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                 Agent-Wechat 架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
```

### 11.6 数据模型

```typescript
// 工作空间
workspaces: { id, name, created_at }

// Agent（用户也是 Agent）
agents: {
  id, workspace_id, role,
  parent_id,        // 父 Agent（组织树）
  llm_history,      // 完整 LLM 对话历史（JSON）
  created_at
}

// 群组（所有对话都是群，P2P = 2人群）
groups: { id, workspace_id, name, context_tokens, created_at }

// 群成员
group_members: {
  group_id, user_id,
  last_read_message_id,  // 已读位置
  joined_at
}

// 消息
messages: {
  id,               // UUID v7（可排序）
  workspace_id, group_id, sender_id,
  content_type, content, send_time
}
```

**关键设计**：
- **IM 与 LLM 历史分离**：`messages` 存 IM 消息，`llm_history` 存完整 LLM 对话（含 tool-call）
- **已读机制**：`last_read_message_id` 追踪已读位置

### 11.7 Agent 运行时

**生命周期**：
```
启动 → 加载 llm_history → 阻塞等待
         ↓
    收到 wake 信号
         ↓
    getAllUnread() 拉取未读
         ↓
    LLM 推理（流式输出）
         ↓
    finish_reason 判断:
    - "tool_calls" → 执行工具 → 继续推理
    - "stop" → 落库 → 回到阻塞等待
```

**内置工具**：

| 工具 | 功能 |
|------|------|
| `create` | 创建子 Agent |
| `send` | 向指定 Agent 发消息 |
| `self` | 获取自身身份信息 |
| `list_agents` | 列出工作空间所有 Agent |
| `list_groups` | 列出可见群组 |
| `send_group_message` | 向群发消息 |
| `send_direct_message` | 向 Agent 私聊 |
| `bash` | 执行 Shell 命令 |

**并发模型**：
- 每个 Agent 一个长驻 Runner
- 同一 Agent 串行处理
- 不同 Agent 可并行
- 事件驱动唤醒，无中央轮询

### 11.8 前端界面

**三区布局**：
```
┌─────────────────────────────────────────────────────────┐
│ 左侧栏                                                   │
│ ├─ Workspace 切换                                        │
│ ├─ 全局搜索                                              │
│ ├─ Agent-Graph 入口                                      │
│ └─ 对话列表（按时间排序，显示未读红点）                    │
├─────────────────────────────────────────────────────────┤
│ 主区                                                     │
│ ├─ 聊天窗口（微信风格消息气泡）                           │
│ └─ Agent 详情面板（LLM history，可折叠）                  │
├─────────────────────────────────────────────────────────┤
│ Agent-Graph 页面（独立路由）                              │
│ └─ 圆点=Agent，箭头=消息流，可点击查看详情                │
└─────────────────────────────────────────────────────────┘
```

**核心特性**：
1. **微信式聊天**：树状多级对话列表，可选择任意 Agent 对话
2. **LLM History 面板**：实时展示 Agent 上下文，Agent 不再是黑箱
3. **流式 Graph**：动态展现蜂群拓扑与实时通信链路
4. **流式 Tool-call**：实时展示工具调用参数

### 11.9 协作模式（Spells）

**Map-Reduce 模式**：
```
[MAP]
1) 将任务拆分为 N 个子任务（N=4 推荐）
2) 为每个子任务创建 Agent，并发送任务

[REDUCE]
3) 等待所有子任务回报
4) 汇总为最终答案
```

**拓扑感知（Topology Skill）**：
每个 Agent 应了解：自己在拓扑中的位置、谁是父 Agent、集体目标是什么。

### 11.10 优势与局限

| 优势 | 局限 |
|------|------|
| **动态拓扑**：运行时创建子 Agent | 依赖 LLM 质量 |
| **任意通信**：Agent 间无层级限制 | 多 Agent 成本高 |
| **人类介入**：微信式 UI 随时介入 | 早期项目，功能迭代中 |
| **透明调试**：LLM History 可见 | 中文社区为主 |
| **极简设计**：只有 create + send | |
