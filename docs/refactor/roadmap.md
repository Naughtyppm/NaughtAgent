# NaughtyAgent 重构路线图

基于 Claude Agent SDK 架构的渐进式重构计划

## 总览

```
Phase 1: 基础设施层对齐 (8-10天) ✅ 当前阶段
    ↓
Phase 2: 工具层重构 (10-12天)
    ↓
Phase 3: Agent 核心层重构 (10-12天)
    ↓
Phase 4: 能力扩展层 (12-15天)
    ↓
Phase 5: 集成层更新 (5-7天)
```

**总计**: 45-56 天（约 2-2.5 个月）

---

## Phase 2: 工具层重构 🔧

**时间**: 10-12 天  
**状态**: ⏸️ 待开始  
**依赖**: Phase 1 完成

### 目标
工具系统符合 MCP 规范，支持动态发现和外部集成

### 主要任务

#### 2.1 工具接口标准化（3天）
- [ ] 统一 Tool 接口（对齐 MCP Tool 定义）
- [ ] 添加 `inputSchema` (JSON Schema)
- [ ] 添加 `outputSchema` (可选)
- [ ] 添加 `icons` 和 `title` 支持
- [ ] 重构 ToolRegistry 支持动态注册

#### 2.2 MCP 客户端实现（4-5天）
- [ ] 实现 `MCPClient` 类
- [ ] 支持 stdio 传输
- [ ] 支持 SSE 传输
- [ ] 实现 `tools/list` 请求
- [ ] 实现 `tools/call` 请求
- [ ] 监听 `tools/list_changed` 事件
- [ ] 实现连接管理和重连

#### 2.3 工具发现机制（2天）
- [ ] 实现 `discoverMCPTools()` 方法
- [ ] 自动包装 MCP 工具为内部 Tool
- [ ] 支持工具热更新
- [ ] 实现工具分页加载

#### 2.4 内置工具优化（2天）
- [ ] 统一内置工具的输入输出格式
- [ ] 添加更详细的错误信息
- [ ] 优化工具执行性能
- [ ] 添加工具执行超时控制

### 交付物
- 符合 MCP 规范的工具接口
- 完整的 MCP 客户端实现
- 工具动态发现系统
- 优化的内置工具
- Phase 2 完成报告

### 关键文件
- `packages/agent/src/tool/tool.ts` - 工具接口
- `packages/agent/src/tool/registry.ts` - 工具注册表

## Phase 3: Agent 核心层重构 🤖

**时间**: 10-12 天  
**状态**: ⏸️ 待开始  
**依赖**: Phase 2 完成

### 目标
主循环和执行模式对齐 Claude Agent SDK

### 主要任务

#### 3.1 Agent 主循环优化（3-4天）
- [ ] 实现标准的 `while(tool_call)` 模式
- [ ] 优化消息历史管理
- [ ] 添加 `max_turns` 限制
- [ ] 实现循环中断和恢复
- [ ] 优化 token 使用

#### 3.2 流式响应实现（2-3天）
- [ ] 实现 `stream()` 方法
- [ ] 定义流式事件类型
- [ ] 支持增量文本传输
- [ ] 支持工具调用通知
- [ ] 实现流式错误处理

#### 3.3 执行模式支持（3-4天）
- [ ] 实现 `ask_llm()` - 简单问答
- [ ] 实现 `run_agent()` - 标准执行
- [ ] 实现 `fork_agent()` - 继承上下文
- [ ] 实现 `run_workflow()` - 工作流编排
- [ ] 添加执行模式选择逻辑

#### 3.4 实时引导队列（2天）
- [ ] 实现 H2A 双缓冲队列
- [ ] 支持暂停/恢复
- [ ] 支持中途注入指令
- [ ] 实现队列优先级

### 交付物
- 优化的 Agent 主循环
- 完整的流式响应系统
- 四种执行模式实现
- 实时引导队列
- Phase 3 完成报告

### 关键文件
- `packages/agent/src/agent/loop.ts` - 主循环
- `packages/agent/src/agent/stream.ts` - 流式响应
- `packages/agent/src/agent/modes.ts` - 执行模式

## 测试策略

### 单元测试
- 每个模块独立测试
- 覆盖率要求：语句 80%+，分支 75%+，函数 85%+，行 80%+

### 集成测试
- Phase 完成后进行集成测试
- 测试模块间交互
- 测试端到端流程

### 回归测试
- 每个 Phase 完成后运行完整测试套件
- 确保不破坏现有功能

### 性能测试
- 对比重构前后的性能指标
- 监控关键路径的延迟
- 优化性能瓶颈

## 风险管理

### 技术风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 破坏现有功能 | 高 | 中 | 保持 API 兼容，完整测试 |
| 性能回归 | 中 | 低 | 性能监控，基准测试 |
| MCP 集成复杂 | 中 | 中 | 参考官方示例，分步实现 |
| 测试覆盖率下降 | 中 | 低 | 强制测试要求，CI 检查 |

### 进度风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 时间估算不准 | 中 | 中 | 预留缓冲时间，灵活调整 |
| 依赖阻塞 | 高 | 低 | 并行任务，提前准备 |
| 需求变更 | 中 | 中 | 敏捷迭代，快速响应 |
