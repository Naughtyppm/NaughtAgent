# Zed AI 设计理念

> 调研时间：2026-01-14
> 参考项目：https://github.com/zed-industries/zed

Zed 是一个高性能的现代代码编辑器（Rust 编写），其 AI 功能设计理念值得借鉴。

## 核心原则

### 1. AI 永不自动运行

- 没有后台监听进程
- 没有文件变动自动触发
- 没有定时任务
- AI 像"工具"而非"守护进程"

### 2. 一切从显式操作开始

- **选区驱动** - 选中代码后触发 AI
- **明确指令** - 用户主动发起请求
- 不做隐式推断，不自作主张

### 3. 人永远在方向盘上

- **Apply/Reject** - 所有修改需人工确认
- 不自动写入文件
- 不自动执行命令
- 用户完全掌控

## 原子化操作

Zed 的 AI 交互是原子化的：

| 操作 | 说明 |
|------|------|
| Inline Assist | 选中代码 → 输入指令 → 预览修改 → 确认/拒绝 |
| Generate | 光标位置 → 描述需求 → 生成代码 → 确认/拒绝 |
| Transform | 选中代码 → 选择转换类型 → 预览 → 确认/拒绝 |

每个操作都是：
```
触发 → 预览 → 确认
```

没有复杂的多步规划，没有自动执行链。

## 与 Agent 模式的对比

| 特性 | Zed 风格 | 典型 Agent |
|------|----------|-----------|
| 触发方式 | 显式（选区/命令） | 可自动 |
| 执行范围 | 单一操作 | 多步任务 |
| 文件修改 | 需确认 | 可自动 |
| 命令执行 | 不支持/需确认 | 可自动 |
| 复杂度 | 简单可控 | 功能强大但风险高 |

## 对 NaughtAgent 的启示

### 采纳的原则

1. **显式触发** - 用户主动调用，无后台进程
2. **人工确认** - 修改必须经用户同意
3. **可预览** - 执行前展示将要做什么

### 平衡点

NaughtAgent 需要在 Zed 的"原子化"和 Claude Code 的"Agent 能力"之间找平衡：

- **保留 Agent 能力** - 支持多步任务、文件操作、命令执行
- **采用 Zed 理念** - 每一步都需确认，不自动执行危险操作
- **分级控制** - 低风险操作可批量确认，高风险操作逐个确认

### 交互设计建议

```
用户发起请求
    ↓
Agent 规划步骤（展示给用户）
    ↓
用户确认/修改计划
    ↓
逐步执行，每步可暂停
    ↓
修改预览 → 用户 Apply/Reject
```

## 总结

Zed 的价值不在于代码实现，而在于**交互理念**：

> 把 AI 当作工具，而不是自主代理。
> 人始终掌控，AI 只是加速器。

这个理念应该贯穿 NaughtAgent 的设计。
