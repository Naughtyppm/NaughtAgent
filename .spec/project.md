# NaughtAgent 项目规范

## 什么是 AI Agent？

**一句话：Agent = LLM + 工具 + 循环**

传统 ChatGPT 只能对话，而 Agent 可以**做事**：

```
传统 LLM：
用户: "帮我创建一个 hello.ts 文件"
LLM:  "好的，你可以用以下命令创建..." （只是告诉你怎么做）

Agent：
用户: "帮我创建一个 hello.ts 文件"
Agent: [调用 write 工具] → 文件已创建 ✅ （真的帮你做了）
```

### Agent 的核心循环

```
┌─────────────────────────────────────────────────────────────┐
│                      Agent Loop                              │
│                                                              │
│   用户输入                                                    │
│      │                                                       │
│      ▼                                                       │
│   ┌─────────┐    "需要读取文件"    ┌─────────┐              │
│   │   LLM   │ ──────────────────→ │  Tool   │              │
│   │  (大脑)  │                     │ (工具)   │              │
│   └─────────┘ ←────────────────── └─────────┘              │
│      │           "文件内容是..."                             │
│      │                                                       │
│      ▼  （重复直到任务完成）                                   │
│   最终回复                                                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键点：**
1. **LLM 决定**调用什么工具（不是写死的流程）
2. **工具执行**后把结果返回给 LLM
3. **循环**直到 LLM 认为任务完成

### 举个例子

用户说："帮我找到项目里所有的 TODO 注释，然后创建一个 todo.md 汇总"

Agent 的执行过程：

```
第 1 轮：
  LLM 思考: "我需要先搜索 TODO"
  LLM 调用: grep({ pattern: "TODO" })
  工具返回: "src/a.ts:10: // TODO: fix bug\nsrc/b.ts:20: // TODO: add test"

第 2 轮：
  LLM 思考: "找到了，现在创建汇总文件"
  LLM 调用: write({ path: "todo.md", content: "# TODO List\n- fix bug (a.ts:10)\n- add test (b.ts:20)" })
  工具返回: "文件已创建"

第 3 轮：
  LLM 思考: "任务完成了"
  LLM 回复: "我已经找到 2 个 TODO 并创建了 todo.md 汇总文件"
```

## Agent 的三种模式

不是所有任务都需要完整的 Agent 循环：

| 模式 | 说明 | 例子 |
|------|------|------|
| **API** | 单次 LLM 调用，不用工具 | "解释这段代码" |
| **Workflow** | 预定义步骤，固定流程 | "/commit" 命令 |
| **Agent** | LLM 自主决定，完整循环 | "重构这个模块" |

### API 模式

```
用户输入 → LLM → 直接回复
```

最简单，一问一答，不调用任何工具。

### Workflow 模式

```
用户输入 → 步骤1 → 步骤2 → 步骤3 → 回复
```

流程是写死的，比如 `/commit`：
1. 执行 `git diff --staged`
2. 让 LLM 生成 commit 消息
3. 执行 `git commit -m "..."`

### Agent 模式

```
用户输入 → LLM决定 → 工具 → LLM决定 → 工具 → ... → 回复
```

LLM 自己决定调用什么工具，最灵活但也最不可预测。

## Agent 的三种类型（配置）

同样是 Agent 模式，但可以有不同的配置：

| 类型 | 可用工具 | 权限 | 用途 |
|------|---------|------|------|
| **build** | 全部 | 写操作需确认 | 默认，干活用 |
| **plan** | 只读 | 禁止写入 | 分析规划用 |
| **explore** | 只读 | 禁止写入 | 快速搜索用 |

```bash
# 默认用 build，可以改文件
naughtagent "帮我写个函数"

# 用 plan，只分析不动手
naughtagent --agent plan "分析这个项目架构"
```

## 项目目标

做一个自己掌控的 Claude Code：

### 核心原则

1. **显式触发** - AI 只在你调用时工作，不会后台偷偷跑
2. **人工确认** - 改文件、跑命令前会问你
3. **可控可维护** - 代码是自己的，想改就改

### 为什么自己做

- 需要定制功能，现有工具不支持
- 想理解 Agent 是怎么工作的
- 不想被闭源工具限制
