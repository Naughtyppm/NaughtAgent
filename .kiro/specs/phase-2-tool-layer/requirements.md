# 需求文档

## 简介

Phase 2 的目标是重构工具层，使其符合 Model Context Protocol (MCP) 规范，支持动态工具发现和外部工具集成。这将使 NaughtyAgent 能够与任何符合 MCP 规范的工具服务器进行交互，大幅扩展其能力边界。

## 术语表

- **Tool（工具）**: Agent 可以调用的功能单元
- **MCP（模型上下文协议）**: Model Context Protocol，用于 AI 系统与外部工具通信的标准协议
- **Tool_Registry（工具注册表）**: 管理所有可用工具的中心化组件
- **MCP_Client（MCP 客户端）**: 负责与 MCP 服务器通信的组件
- **Tool_Schema（工具模式）**: 描述工具输入输出格式的 JSON Schema
- **Stdio_Transport（标准输入输出传输）**: 通过 stdin/stdout 进行进程间通信的方式
- **SSE_Transport（SSE 传输）**: Server-Sent Events 传输，通过 HTTP SSE 进行通信的方式
- **Tool_Discovery（工具发现）**: 自动检测和注册可用工具的过程
- **Hot_Reload（热重载）**: 在运行时动态更新工具列表的能力

## 需求列表

### 需求 1: 工具接口标准化

**用户故事：** 作为开发者，我希望所有工具遵循统一的接口规范，以便能够一致地定义、注册和调用工具。

#### 验收标准

1. 工具接口必须包含 `id`、`description`、`inputSchema` 和 `execute` 字段
2. 工具接口应支持可选的 `outputSchema`、`title` 和 `icons` 字段
3. 当定义工具时，系统应根据 JSON Schema 规范验证 `inputSchema`
4. 工具注册表应支持运行时动态注册工具
5. 当注册工具时，工具注册表应验证工具 ID 的唯一性

### 需求 2: MCP 客户端实现

**用户故事：** 作为系统架构师，我希望实现符合 MCP 规范的客户端，以便能够与任何 MCP 服务器进行通信。

#### 验收标准

1. MCP 客户端必须支持 stdio 传输方式用于基于进程的通信
2. MCP 客户端必须支持 SSE 传输方式用于基于 HTTP 的通信
3. 当连接到 MCP 服务器时，MCP 客户端应发送 `tools/list` 请求以发现可用工具
4. 当调用工具时，MCP 客户端应发送包含工具名称和参数的 `tools/call` 请求
5. 当 MCP 服务器发出 `tools/list_changed` 事件时，MCP 客户端应刷新工具列表
6. 如果连接 MCP 服务器失败，则 MCP 客户端应实施指数退避重试策略
7. MCP 客户端应维护连接状态并支持优雅的重连

### 需求 3: 工具发现机制

**用户故事：** 作为用户，我希望系统能够自动发现和加载可用的工具，无需手动配置每个工具。

#### 验收标准

1. 当系统启动时，系统应自动从配置的服务器发现所有 MCP 工具
2. 当发现新的 MCP 工具时，系统应将其包装为内部工具实例
3. 当接收到 `tools/list_changed` 事件时，系统应在不重启的情况下更新工具列表
4. 当工具列表很大时，系统应支持分页加载工具
5. 系统应在每个 MCP 服务器 500ms 内完成工具发现

### 需求 4: 内置工具优化

**用户故事：** 作为开发者，我希望内置工具（Read, Write, Edit, Bash, Glob, Grep）能够提供一致的接口和更好的错误处理。

#### 验收标准

1. 内置工具必须符合标准化的工具接口
2. 当工具执行失败时，系统应返回包含错误类型和上下文的结构化错误信息
3. 系统应对所有工具强制执行超时，默认值为 30 秒
4. 在适用的情况下，内置工具应为长时间运行的操作提供进度反馈
5. 内置工具应在执行前使用 Zod schema 验证输入参数

### 需求 5: 向后兼容性

**用户故事：** 作为现有用户，我希望系统升级后，现有的工具调用代码能够继续正常工作。

#### 验收标准

1. 系统必须保持与现有工具定义的向后兼容性
2. 当使用旧版工具格式时，系统应自动将其转换为新格式
3. 系统应在过渡期间同时支持新旧工具注册 API
4. 当使用旧版 API 调用工具时，系统应发出弃用警告但继续执行

### 需求 6: 类型安全

**用户故事：** 作为开发者，我希望工具系统提供完整的 TypeScript 类型支持，以便在编译时捕获错误。

#### 验收标准

1. 工具接口必须使用 TypeScript 完全类型化
2. 系统应使用 Zod 进行运行时 schema 验证
3. 当定义工具时，系统应从 Zod schema 推断 TypeScript 类型
4. MCP 客户端应为所有 MCP 协议操作提供类型化方法
5. 工具注册表应提供类型安全的工具查找方法

### 需求 7: 性能要求

**用户故事：** 作为用户，我希望工具调用的开销尽可能小，不影响整体响应速度。

#### 验收标准

1. 工具发现应在合理时间内完成：本地 stdio 服务器 500ms 内，远程 SSE 服务器 2000ms 内
2. 工具注册表查找应在 10ms 内完成任何工具的查找
3. MCP 客户端应为工具执行增加少于 100ms 的开销（不包括实际工具执行时间）
4. 系统应缓存工具 schema 以避免重复解析
5. 当顺序调用多个工具时，系统应复用 MCP 连接
6. 工具发现支持异步模式，不阻塞 Agent 启动

### 需求 8: 错误处理

**用户故事：** 作为开发者，我希望工具执行失败时能够获得清晰的错误信息，便于调试和恢复。

#### 验收标准

1. 当工具执行失败时，系统应返回包含 `type`、`message` 和 `context` 字段的错误对象
2. 如果 MCP 服务器不可达，则系统应返回包含重试建议的 `ConnectionError`
3. 如果工具参数验证失败，则系统应返回包含详细字段级错误的 `ValidationError`
4. 如果工具执行超时，则系统应返回包含已用时间信息的 `TimeoutError`
5. 系统应使用适当的严重级别记录所有工具错误

### 需求 9: 测试覆盖率

**用户故事：** 作为质量保证工程师，我希望工具系统有充分的测试覆盖，确保功能正确性和稳定性。

#### 验收标准

1. 系统应达到工具相关代码最低 80% 的语句覆盖率
2. 系统应达到工具相关代码最低 75% 的分支覆盖率
3. 系统应达到工具相关代码最低 85% 的函数覆盖率
4. 系统应达到工具相关代码最低 80% 的行覆盖率
5. 系统应包含 MCP 客户端通信的集成测试

### 需求 10: 配置管理

**用户故事：** 作为系统管理员，我希望能够通过配置文件管理 MCP 服务器连接和工具设置。

#### 验收标准

1. 系统应支持 MCP 服务器端点的配置文件
2. 配置应同时支持 stdio 和 SSE 传输配置
3. 当配置更改时，系统应支持无需重启的热重载
4. 系统应在加载时验证配置 schema
5. 如果配置无效，则系统应记录错误并使用默认设置
